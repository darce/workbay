# Security Heuristics Lexicon

## About this document

> **Referenced, not read end-to-end.** A dedicated rubric beside engineering, business/marketing, design/aesthetics, writing, and accessibility. It governs **security as a spanning concern**: the trust boundaries, injection sinks, privilege grants, and misconfigurations that turn a working feature into an exploitable one. Like accessibility, security cuts across the other lexicons — an SQL-injection rule is also a data rule, a least-privilege rule is also an architecture rule — so it earns its own lexicon rather than scattering.

> **Provenance — grounding in progress.** **§1 (AI/ML & agent security)** is sourced to Steve Wilson, *The Developer's Playbook for Large Language Model Security* (`llm-security-playbook`), extended to adversarial ML — poisoning, backdoors, evasion, ML supply chain — by Sotiropoulos, *Adversarial AI* (`sotiropoulos-adversarial-ai`). **§2 (web application security)** has been **re-grounded** against the canonical web-security texts: the browser-model rules (XSS, CSRF, cookies, CSP/HSTS, redirects, TLS, clickjacking, CORS, content sniffing, `postMessage`) to Michal Zalewski, *The Tangled Web* (`zalewski-tangled-web`), and the server-side rules (SQL/OS/LDAP/XXE injection, SSRF, path traversal, broken access control, sessions, authentication) to Stuttard & Pinto, *The Web Application Hacker's Handbook* (`stuttard-wahh`). A few WEB rows those two texts do not establish — mass assignment [WEB-19], native deserialization [WEB-20], upload hardening [WEB-12], secrets-in-VCS [WEB-16] — remain **`bootstrap`**. Several **§3 (PHP)** rows are grounded in *Modern PHP Security* (Chauchefoin & David, sec4dev 2020). **§§4–5 (WordPress, PostgreSQL)** remain a **`bootstrap`** pass — the corpus holds no WordPress- or PostgreSQL-specific security text, so those rows are drawn from established practice and held at tier **S/J**, not settled Blockers-by-citation. **§6 (secure design principles)** — least privilege, complete mediation, minimize the trusted computing base, fail-safe defaults, defense in depth, threat modelling — is grounded in Anderson, *Security Engineering* (`anderson-security-engineering`). Treat any `bootstrap` row as a strong default under review.

**Scope.** AI/ML and agent security, general web application security, secure-design principles, and the PHP / WordPress / PostgreSQL stack. It does **not** yet cover cryptography, network security, OS hardening, or identity protocols in depth — those await both the sources and the pass.

Row format matches the sibling lexicons: `ID | Trigger | Rule | Answers | T·P | Src` — tier **B**locker (exploitable, data-loss, or auth-bypass if violated) / **S**hould (strong default) / **J**udgment; phase **p**lan (architecture/threat-model) / **w**rite (code) / **r**eview (config/audit). `Src` names the distilled source, or `bootstrap` for a not-yet-sourced row. IDs are stable citation keys (`[WEB-01]`) and immutable once assigned. Cross-lexicon borders cite `↔ eng/biz` rules instead of restating them.

## 1. AI/ML & Agent Security

> `SEC-01..10` (LLM/agent) are sourced to Wilson, *The Developer's Playbook for Large Language Model Security* (`llm-security-playbook`; moved here from engineering — existing `[SEC-02]` citations still resolve). `SEC-11..16` extend the family to **adversarial ML** — training-data poisoning, backdoors/trojans, evasion, and the ML supply chain — from Sotiropoulos, *Adversarial AI* (`sotiropoulos-adversarial-ai`).

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| SEC-01 | Data crossing any component boundary (user↔app, app↔LLM, LLM↔services) | **Validate at every trust boundary** — trust changes at each crossing; the LLM's output side is a boundary too (↔ [WEB-08] object access the route check does not enforce; ↔ [WEB-21] a frame that delivers clicks the eye never chose; ↔ [PG-02] tenant isolation left to every handler's WHERE; ↔ grph [GRPH-18] a borderline match that asserts identity; ↔ ux [PERC-03] a group the layout implies but the code does not wire; ↔ ux [HAI-05] autonomy named beyond the tested envelope) | Where does trust change, and what validation runs there — in and out? | S·p | llm-security-playbook ch-3 |
| SEC-02 | LLM output passed to SQL, shell, eval, or rendered as HTML | **Treat LLM output as untrusted input** — generated text can carry injected payloads into any interpreter (↔ [WEB-01], [WEB-04] generated text is untrusted input at every interpreter sink) | Is the output parameterized, escaped, or encoded before the sink? | B·r | llm-security-playbook ch-7 |
| SEC-03 | User or external content interpolated into an LLM prompt | **Delimit untrusted prompt segments** — the model can't distinguish data from instructions unless the boundary is marked | Is every untrusted segment structurally marked and out of instruction position? | S·w | llm-security-playbook ch-4 |
| SEC-04 | LLM granted a tool, plugin, API scope, or DB permission | **Least-privilege agency** — a jailbroken model exercises every permission it holds (↔ [PG-01]; ↔ biz [BOOT-07] one landlord's hold is fatal when there is no second exit) | Is this the minimum scope the feature needs? | B·p | llm-security-playbook ch-7 |
| SEC-05 | LLM can trigger an irreversible, financial, or destructive action | **Human-in-the-loop gate** — autonomy plus prompt injection equals attacker-driven transactions (↔ [SEC-04] broad scope is the irreversible side of the same trade; ↔ biz [CLM-02] the aggressive phrasing is the underpriced irreversible claim; ↔ eng [RLSE-08] write the rollback before ship, not at 2am) | Does a human approve before the side effect executes? | B·p | llm-security-playbook ch-7 |
| SEC-06 | Secrets or PII in a prompt template, fine-tune set, or RAG store | **Don't teach the model what it must never say** — anything the LLM can read can be extracted by any user (↔ [WEB-16] a secret the model was taught is a secret the user can extract) | What happens if this datum is disclosed verbatim? | B·p | llm-security-playbook ch-5 |
| SEC-07 | Web pages, files, or DB rows fed into a RAG prompt | **Retrieved content is attacker content** — indirect injection rides in on documents | Is retrieved text screened and confined to a data role? | S·w | llm-security-playbook ch-4 |
| SEC-08 | Public LLM endpoint or feature without usage limits | **Rate-limit and cap per caller** — unmetered access invites DoS and denial-of-wallet (↔ [WEB-17] unmetered public endpoints are DoS and denial-of-wallet by default) | Are per-caller limits, token caps, and budget alerts configured? | S·w | llm-security-playbook ch-8 |
| SEC-09 | LLM-suggested package added to a dependency manifest | **Verify hallucinated dependencies** — attackers register malware under names models invent (↔ eng [AGT-02] a name that was never resolved is not a referent) | Does this package exist with the expected owner and real history? | B·r | llm-security-playbook ch-6 |
| SEC-10 | Model or dataset pulled from a hub at build/run time | **Pin and record provenance** — unpinned weights are an unauditable dependency (↔ [SEC-09] a model-invented package name is not a referent until it resolves; ↔ grph [GRPH-14] every artifact must point at the inputs that produced it; ↔ writ [WRIT-26] name the source or cut the claim; ↔ epi [FORE-06] a point probability with no decomposition has no walkable ancestors; ↔ a11y [A11Y-27] authority is the primary text, not a derived summary; ↔ writ [WRIT-41] a citation that cannot be followed is worse than none) | Is the exact revision pinned and integrity verified? | S·w | llm-security-playbook ch-9 |
| SEC-11 | A training or fine-tune job reads data from the web, users, vendors, or a continuous live feed | **Poison-gate training data** — data crossing a trust boundary into training needs lineage and versioning, statistical anomaly gates, *and* model-aware poison probes (activation clustering / spectral signatures / **RONI**), because **data poisoning** — including **clean-label** poison that passes label QA and slips statistical outlier checks — rewrites model integrity before any inference defense applies (↔ [MLDATA-04] data crossing a trust boundary into training needs lineage before the model absorbs it) | Who can write rows this model will learn from, and does the gate run model-aware poison probes, not just label QA? | B·p | sotiropoulos-adversarial-ai |
| SEC-12 | Accepting third-party weights, or promoting a model to the registry | **Probe for backdoors before promote** — run trigger/canary and activation/spectral probes (and **RONI** where the training set's influence is still available) on accepted weights, because clean test accuracy does not reveal a **backdoor/trojan** that flips class under a fixed patch (↔ eng [TEST-15] a green suite that never exercised the failure certifies nothing; ↔ ml [COST-04] utilization green while accepted output is not; ↔ biz [OPS-01] the metric pays for the 2am behavior, not the one you meant; ↔ biz [AIPX-02] offline score up while acceptance stays flat) | Would this model misclassify under a fixed trigger while passing the standard test set? | B·r | sotiropoulos-adversarial-ai |
| SEC-13 | An inference loader deserializes model files (pickle / custom ops / `Lambda` layers) | **Model files are code** — load weights only from a signed, allowlisted source in a non-executable format (e.g. safetensors), scan for non-allowlisted layers, and verify the architecture against an approved card, because a **trojan** payload needs no poisoned training set (↔ [WEB-20] deserializing untrusted model bytes is the same RCE class as untrusted objects) | Can loading this artifact execute code or branch on a hidden trigger? | B·w | sotiropoulos-adversarial-ai |
| SEC-14 | A classifier is exposed on a public or attacker-reachable inference API (vision / NLP / audio) | **Evasion budget in the eval gate** — put a documented **adversarial-example** eval (FGSM/PGD-class; patches / text as the domain demands) with a max attack-success rate in CI, and add adversarial training or layered defenses when the harm is high (safety, fraud, auth, moderation), because clean metrics hide inference-time **evasion** (≠ [EVAL-06], which measures production noise, not an adaptive adversary) | What attack suite and acceptable attack-success rate are in CI for this threat model? | S·p | sotiropoulos-adversarial-ai |
| SEC-15 | Train or serve resolves packages, base images, or models from a public `pip` / hub / `from_pretrained` | **Pin the ML supply chain** — use allowlisted private indexes, digest-pinned images, and an AI/ML SBOM, and re-eval + poison-detect any transfer-learning checkpoint, because a poisoned package or upstream checkpoint bypasses in-house data QA; pin the model/dataset revision itself under [SEC-10] (↔ [EMB-05] an unpinned package, image, or checkpoint is a silent supply-chain rewrite) | Which index, image, and checkpoint are pinned, and is the transfer-learning base re-evaluated? | B·w | sotiropoulos-adversarial-ai |
| SEC-16 | Identities can mutate training data, the feature store, or the model registry | **Least privilege on the ML control plane** — split train / promote / serve rights and monitor writes to anything that becomes production truth, because a classic **least-privilege** failure enables poison and trojan with no novel math (↔ [SEC-04]; [Principle 14] both shrink what a single compromised identity can reach) | Which human or job can silently replace the production weights or 1% of the labels? | B·p | sotiropoulos-adversarial-ai |
| SEC-17 | A capture-facing identity system reports only identity match rates (true-accept, false-match) with no attack-presentation error table | **Attack rates are not match rates** — when live captures drive identity decisions, report presentation-attack detection error separately from identity error — artefacts accepted as genuine, and genuine presentations rejected — at a declared detection threshold and per named attack class, because a true-accept rate is measured against cooperative subjects and says nothing about how often a print, replay, or mask is accepted (↔ [SEC-14] the same clean-metric blindness, digital perturbation rather than physical presentation) | Are attack-presentation error rates reported separately from identity false-match rates, at a declared detection threshold? | S·r | handbook-face-recognition |
| SEC-18 | An accept fires on similarity score while the liveness check runs after the accept, is logged only, or is fused with no written policy | **The liveness verdict must gate the accept** — when a live capture can be a print, replay, or worn mask, the presentation-attack stage must reject before the gallery comparison commits an accept; if the two scores are fused instead, the fusion policy and both thresholds must be written down and testable, because an attacker holding an enrolled person's photograph scores as that person, and a liveness signal nothing blocks on is telemetry, not a control | Can a presented photograph or replayed video of an enrolled person reach a final accept on similarity alone? | B·w | handbook-face-recognition |
| SEC-19 | Liveness is an interactive motion challenge (blink, smile, turn head) and the attack evaluation set holds only flat printed photographs | **A motion challenge is not an attack set** — when liveness rests on interactive motion, extend the evaluation set to replayed video and to partial or three-dimensional artefacts (cut-out regions, worn masks, attack eyewear) before the ship gate passes, because a recording of the enrolled person performing the requested action satisfies a motion challenge, and a region-specific check is defeated by an artefact covering only that region | Does the attack set include replayed video and partial or three-dimensional artefacts, or only static print? | S·p | handbook-face-recognition |
| SEC-20 | Presentation-attack detection is certified on the same camera, lighting, and capture protocol it was trained on, while the product ships to varied client devices | **Certify liveness on held-out capture hardware** — when client sensors, resolution, or lighting differ from the training capture, report detection error under a leave-one-device or leave-one-capture-source protocol, because cross-source error degrades sharply under illumination, resolution, and sensor-noise shift, so a matched-domain number certifies the lab and the first unseen camera is a field bypass rather than a regression (↔ ml [FAIR-05] capture conditions are evaluation factors — this is the held-out form of that factor) | Is there a leave-one-device table, or only same-camera lab numbers? | S·r | handbook-face-recognition |

## 2. Web Application Security

> Re-grounded against Zalewski, *The Tangled Web* (`zalewski-tangled-web`, browser security model) and Stuttard & Pinto, *The Web Application Hacker's Handbook* (`stuttard-wahh`, server-side attacks). The four rows those texts do not establish — [WEB-12], [WEB-16], [WEB-19], [WEB-20] — remain `bootstrap`.

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| WEB-01 | User-controlled data concatenated into an SQL string (or interpolated into a query without a binder) | **Parameterize every query** — string-built SQL lets any quote/terminator break out into code; use prepared statements + bound params, including for values re-read from the database (second-order injection); binders bind values, so dynamic identifiers need [WEB-35] (↔ eng [DATA-12] both fail when a value is interpolated into a query instead of bound) | Is every SQL value bound — including stored values re-used in a later query? | B·w | stuttard-wahh |
| WEB-02 | User input passed to `exec`/`system`/`shell_exec`/`Runtime.exec`/`ProcessBuilder` without an argv array | **Never shell-interpolate untrusted input** — metacharacters become a second command; use argv APIs or refuse shell (↔ eng [DATA-12] never interpolate untrusted bytes into an interpreter) | Is the OS command built as discrete args with no shell? | B·w | stuttard-wahh |
| WEB-03 | LDAP filter built with string concat of request fields | **Allowlist LDAP filter input** — `)(uid=*` style input widens the filter and bypasses authz; permit only simple allowlisted characters and reject the filter metacharacters — parentheses, `*`, `&`, `=`, pipe — and null bytes, or use a typed directory API (WAHH's stance is reject, not sanitize-in-place) (↔ eng [DATA-12] metacharacter breakout from data into control plane) | Is every LDAP filter value allowlisted and metacharacter-rejected? | B·w | stuttard-wahh |
| WEB-04 | Response body echoes request data (or stored user content) into HTML without context-aware encoding | **Encode for the sink context** — unescaped HTML lets scripts run as the victim (reflected/stored XSS) | Is every untrusted string escaped for HTML/attr/JS/URL before emit? | B·w | zalewski-tangled-web |
| WEB-05 | Client-side code writes `location`/`document.write`/`innerHTML` from `location.hash`/`postMessage`/URL params | **Treat DOM sinks as XSS sinks** — client-only flows still execute attacker script in the origin | Is untrusted data never assigned to HTML/JS sinks without sanitization? | B·w | zalewski-tangled-web |
| WEB-06 | State-changing POST/PUT/DELETE lacks a per-session CSRF token verified server-side (`SameSite` cookies are defense-in-depth, not a substitute) | **Require anti-CSRF on mutations** — a third-party site can forge the browser's cookies into a state change | Does every state change verify a CSRF token or equivalent origin binding? | B·w | zalewski-tangled-web |
| WEB-07 | Server fetches a URL/host derived from user input (webhooks, previews, importers) | **Block SSRF to internal nets** — attacker-chosen URLs reach link-local/metadata/admin services; allowlist destinations or deny private ranges after re-resolve, and treat the fetched response body as untrusted (never reflect it raw into HTML/JSON) | Is the fetch target allowlisted or proven non-internal, and the returned body untrusted? | B·w | stuttard-wahh |
| WEB-08 | Object ID / path param used in a query without comparing ownership or tenant to the authenticated principal | **Authorize the object, not just the route** — knowing another ID is enough for IDOR/data disclosure (↔ eng [ARCH-13] a property left to handler vigilance is not enforced; ↔ biz [CLM-03] a policy sentence the implementation does not hold; ↔ ml [PROV-05] a prediction an undeclared consumer will treat as a decision; ↔ eng [TEAM-01] an architectural split the org does not communicate across; ↔ eng [NAME-03] a name that promises a type or behavior the code does not hold) | Does the handler enforce ownership/tenant before read or write? | B·w | stuttard-wahh |
| WEB-09 | Session ID accepted from URL, or session not regenerated on privilege change (login) | **Rotate session ID on auth** — fixation reuses a pre-set SID after login as the victim | Is a new session ID issued at login and the SID never taken from the URL? | B·w | stuttard-wahh |
| WEB-10 | Auth cookie set without `Secure`+`HttpOnly` on a TLS site | **Lock down cookie flags** — set `Secure`+`HttpOnly` and a tight domain/path scope (Zalewski ch.9); missing flags enable theft via XSS or leakage over HTTP. `SameSite` is a later addition (defense-in-depth, not from this source) | Are session cookies `Secure` and `HttpOnly` (and `SameSite` where supported)? | S·r | zalewski-tangled-web |
| WEB-11 | Password login for privileged accounts with no second factor or step-up | **Multi-factor / step-up for privileged auth** — require a second factor and step-up re-authentication for admin and high-risk paths; password-only admin falls to phishing and reuse, and a token alone won't stop a live attacker (↔ [WEB-34] step-up per action) | Is a second factor or step-up reauth required for admin and high-risk actions? | S·p | stuttard-wahh |
| WEB-12 | Upload accepts arbitrary extension/content-type and stores under a web-served path | **Store uploads outside exec root** — executable or HTML uploads become RCE or stored XSS | Is the upload non-executable, type-validated, and outside the docroot? | B·w | owasp-web-hardening §file-upload |
| WEB-13 | User path segment joined to a base directory without canonicalization + prefix check | **Reject path traversal** — `../` escapes the intended root and reads secrets | Is the resolved path verified to stay under the allowed base? | B·w | stuttard-wahh |
| WEB-14 | Redirect target (`?next=`, `return_url`) taken from the request without a same-origin/allowlist check | **Allowlist redirect targets** — off-site redirects turn the app into a phishing trampoline; require same-site or an explicit allowlist and reject non-`http(s)` schemes (`data:`/`javascript:` inherit the origin) | Is the redirect target same-site or allowlisted, and non-pseudo-URL? | S·w | zalewski-tangled-web |
| WEB-15 | App responses omit CSP and HSTS (or set weak/default headers) on HTTPS endpoints | **Ship CSP + HSTS** — missing headers leave XSS impact high and enable SSL strip | Are CSP and HSTS present and non-permissive on all HTTPS responses? | S·r | zalewski-tangled-web |
| WEB-16 | API keys, passwords, or private keys appear in source, committed config, or client bundles | **Never commit secrets** — repo/history exposure is permanent credential theft (↔ [SEC-06] a secret in repo or prompt is extractable by anyone who can read the channel) | Are secrets only in a secret store or env, never in VCS or client code? | B·r | owasp-web-hardening §secrets |
| WEB-17 | Auth, password-reset, or expensive endpoints have no per-IP/account rate limit | **Rate-limit abuse endpoints** — unbounded attempts enable credential stuffing and DoS (↔ [SEC-08] unbounded auth and expensive routes are the same abuse surface as unmetered LLM calls) | Are login/reset/OTP routes rate-limited and lockout-aware? | S·w | stuttard-wahh |
| WEB-18 | HTTPS page loads active `http:` subresources, or admin/login served over cleartext | **No mixed content; ship HSTS** — load only HTTPS active subresources, send `HSTS`, set `Secure` cookies, and allow no cleartext admin, or active MITM replaces script and owns the origin (a TLS-version/cipher floor and HSTS preload are modern additions beyond this source) | Any `http:` active subresource or cleartext admin, and is HSTS set? | B·r | zalewski-tangled-web |
| WEB-19 | Framework mass-assigns request body fields onto a model/entity | **Allowlist assignable model fields** — bind only an explicit safe field list when mapping a request body onto an entity; an extra key like `role` writes a protected attribute (mass assignment) (authz *flags* are [WEB-32]; WordPress is [WP-14]) | Is the binder restricted to an explicit safe field list? | B·w | owasp-web-hardening §mass-assignment |
| WEB-20 | `unserialize`/`pickle`/`ObjectInputStream`/binary object decode on untrusted bytes | **Refuse untrusted deserialization** — gadget chains turn bytes into RCE (↔ [PHP-02] native object decode of untrusted bytes is RCE via gadget chains) | Is untrusted input never deserialized with a native object format? | B·w | owasp-web-hardening §deserialization |
| WEB-21 | A state-changing or authenticated UI can be framed cross-origin (no frame-denial) | **Deny cross-origin framing** — send `X-Frame-Options: DENY`/`SAMEORIGIN` or CSP `frame-ancestors` on sensitive UI, or an opaque overlay delivers the victim's clicks with ambient cookies (**clickjacking / UI redress**) (↔ [WEB-08] ambient authority past an unenforced object boundary) | Can an attacker's page iframe this action UI? | S·r | zalewski-tangled-web |
| WEB-22 | User-controlled bytes served as `text`/`json`/`csv`/unknown type without `nosniff` | **Stop content sniffing** — set the correct `Content-Type` plus `X-Content-Type-Options: nosniff`, and prefer `Content-Disposition: attachment` for downloads, or a browser may render the payload as HTML and run script | Could a browser sniff this response as HTML? | S·w | zalewski-tangled-web |
| WEB-23 | `Access-Control-Allow-Origin: *` or a reflected `Origin` on a credentialed/private API | **Allowlist CORS origins** — echo only exact allowlisted origins and never pair `*` with credentials, or CORS hands attacker pages the responses the same-origin policy would have hidden | Can another origin read this body with the user's cookies? | S·r | zalewski-tangled-web |
| WEB-24 | A `postMessage` handler checks `event.origin` with substring/`endsWith`, or not at all | **Verify message origin exactly** — compare `event.origin` with `===` against an allowlist, or a fuzzy match accepts `trusted.com.attacker.tld` (↔ grph [GRPH-18] a borderline match must not assert identity) | Is the message origin compared exactly to a fixed list? | S·w | zalewski-tangled-web |
| WEB-25 | Request data flows into a response header (`Location`, `Set-Cookie`, custom) | **Strip CR/LF from header values** — never build a header by concatenating untrusted input; newline injection yields header splitting, XSS, and cache poisoning | Is any response-header value built from request data? | B·w | zalewski-tangled-web |
| WEB-26 | An XML parser processes untrusted XML with DTD / external-entity resolution enabled | **Disable XML external entities** — turn off DTD and external-entity resolution on untrusted XML, or `SYSTEM` entities read local files and pivot to SSRF (**XXE**; ≠ [WEB-20] — XML entity resolution, not native object decode) | Are DTDs and external entities disabled on every untrusted XML parser? | B·w | stuttard-wahh |
| WEB-27 | The server builds a back-end request or query from raw user strings (or duplicate params) | **Delimit back-end parameters** — encode and allowlist the parameter set for a server-built request; a duplicate or injected key overrides the intended value (**HTTP parameter pollution**) | Can a user inject `&role=admin` or a duplicate key into a back-end call? | S·w | stuttard-wahh |
| WEB-28 | Login / register / reset responses differ by whether the username exists (status/body/timing) | **Uniform auth responses** — make valid- and invalid-username responses identical in content and timing, or the endpoint leaks a target list (**username enumeration**) | Do real- and fake-user auth responses match aside from nonces? | S·r | stuttard-wahh |
| WEB-29 | Session identifiers are sequential, structured, or otherwise low-entropy | **Unpredictable session tokens** — generate high-entropy, meaningless session IDs, or predictable tokens let an attacker mint another user's session | Given a sample of session IDs, can the next be forecast or mapped to a user? | B·w | stuttard-wahh |
| WEB-30 | Logout only clears the client cookie; the server session stays valid; no idle expiry | **Terminate sessions server-side** — invalidate server session state on logout and idle timeout, not just the client cookie, or a hijacked SID stays live after "logout" | After logout, does replaying the old SID still authorize? | B·w | stuttard-wahh |
| WEB-31 | A privileged URL or method works for a low-privilege or anonymous caller (the UI merely hides it) | **Enforce authorization on every function** — check the required role on the server for each URL and method; hiding a link or trusting a client `isAdmin` flag is not access control (**vertical privilege escalation**; ↔ [WP-03] for the WordPress capability check) | Can a normal session reach an admin URL or a method only referenced in client JS? | B·w | stuttard-wahh |
| WEB-32 | An access decision reads a role/level from a request parameter, hidden field, or cookie (`admin=true`) | **Never authorize from client-supplied flags** — load authority from server-side session state only; request parameters are attacker-controlled (**parameter-based access control**; ↔ [WEB-19] mass-assigned model fields, [WP-14] WordPress role fields) | Does flipping `admin=true` or a role cookie change what the user can do? | B·w | stuttard-wahh |
| WEB-33 | Authorization checks are copy-pasted per handler; some paths are unchecked | **Deny-by-default authorization on every path** — enforce access control through shared, session-driven, deny-by-default checks covering every sensitive handler (one module or equivalent middleware), or piecemeal per-handler checks miss paths and rot under change | Is there any code path that serves sensitive data without the shared deny-by-default authz check? | S·p | stuttard-wahh |
| WEB-34 | A high-impact action (add payee, transfer, role change) proceeds on the ambient session alone | **Re-authenticate for critical actions** — require fresh reauth or dual authorization before an irreversible or high-value mutation, so a stolen session or CSRF cannot complete it | Can a critical mutation complete with only the existing session cookie? | S·w | stuttard-wahh |
| WEB-35 | Dynamic SQL identifier (table / column / `ORDER BY` / sort direction) built from request input | **Allowlist SQL identifiers** — a bound parameter binds a *value*, so a table, column, or sort direction taken from user input must be mapped through a fixed allowlist, or it is injection no binder can stop (↔ [WEB-01] identifiers are not values — binders cannot parameterize a table or column name) | Is every dynamic identifier/sort mapped through an allowlist, not concatenated? | B·w | stuttard-wahh |

## 3. PHP Security

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| PHP-01 | Authz/authn or token compare uses `==`/`!=` between request data and secrets/IDs | **Use strict comparison** — type juggling (`0e…` magic hashes, `"0"==false`) bypasses equality checks; prefer `===`/`hash_equals` | Are all security-sensitive compares strict (`===`/`hash_equals`)? | B·w | php-security §strict-comparison |
| PHP-02 | `unserialize()` called on cookies, POST bodies, cache blobs, or queue payloads | **Never unserialize untrusted data** — POP gadget chains in app/framework classes become RCE (↔ [WEB-20] untrusted object decode is RCE regardless of language) | Is every `unserialize` removed or limited to signed input + `allowed_classes`? | B·w | Modern PHP Security |
| PHP-03 | Filesystem API (`file_exists`, `file_get_contents`, `getimagesize`, …) takes a user-influenced path | **Block phar:// side-effects** — path wrappers deserialize PHAR metadata and run object chains without calling `unserialize` | Are user paths blocked from `phar://` and other dangerous wrappers? | B·w | Modern PHP Security |
| PHP-04 | `include`/`require` path built from request params or remote URLs | **Pin include roots** — user-controlled includes are LFI/RFI and code execution | Is the include path from a fixed map, never raw user input? | B·w | php-security §include-paths |
| PHP-05 | `extract($_GET)` / `extract($_POST)` / `parse_str` into the current symbol table without `EXTR_SKIP` | **Do not extract request into scope** — attackers overwrite existing variables (register_globals class) | Is `extract` on request data absent or forced to non-overwrite flags? | B·w | php-security §extract |
| PHP-06 | `preg_replace` with the `/e` modifier (or a dynamic pattern from user input) | **Ban preg_replace /e** — the replacement is evaluated as PHP code | Is `/e` absent and are patterns never user-controlled? | B·w | php-security §preg-replace-e |
| PHP-07 | Production `php.ini` leaves `exec`/`system`/`passthru`/`shell_exec`/`proc_open` enabled without need | **Disable unused dangerous functions** — post-RCE or LFI impact collapses if dangerous symbols are removed (↔ ml [SERVE-03] an unused live path fires catastrophically when something else fails) | Is `disable_functions` set and verified against actual app needs? | S·r | Modern PHP Security |
| PHP-08 | Superglobals used raw for business decisions without `filter_input`/`filter_var` or allowlist validation | **Validate at the boundary** — untyped input reaches sinks; filter or schema-validate before use (↔ [SEC-01] every trust boundary re-validates — superglobals are one crossing) | Is every external input validated to type/range before logic or sinks? | S·w | php-security §input-validation |
| PHP-09 | Passwords stored or checked with `md5`/`sha1`/`crypt` without a modern KDF | **Use password_hash/password_verify** — fast hashes are offline-crackable; `password_hash` uses adaptive algorithms | Are all password stores using `password_hash` (argon2/bcrypt)? | B·w | php-security §password-hash |
| PHP-10 | App can read paths outside its tree; `open_basedir` unset in shared/hosting PHP | **Constrain open_basedir** — LFI and path bugs cannot escape the allowed tree | Is `open_basedir` set to app + upload dirs only in prod? | S·r | Modern PHP Security |
| PHP-11 | Echo/`<?=` of request or DB strings into HTML without `htmlspecialchars(..., ENT_QUOTES, 'UTF-8')` or template auto-escape | **Escape every HTML emit** — PHP defaults do not encode output; XSS follows (↔ [WEB-04] the sink context decides the encoding, not the language default) | Is all HTML output escaped (or auto-escaped by the template engine)? | B·w | php-security §html-escape |
| PHP-12 | Upload handler trusts `$_FILES['type']` or the client filename extension alone | **Verify content, not client MIME** — spoofed types upload PHP polyglots; check extension allowlist + content sniff + no-exec storage | Is the extension allowlisted and the file stored non-executable outside docroot? | B·w | Modern PHP Security |
| PHP-13 | Class name for `new $class(...)` taken from routing/request without an allowlist | **Allowlist dynamic instantiation** — attacker-chosen classes run dangerous constructors | Is the instantiable class set a fixed allowlist? | B·w | Modern PHP Security |
| PHP-14 | Prod config has `display_errors=On` or broad `error_reporting` printed to clients | **Hide errors from clients** — stack traces leak paths, queries, and secrets; log instead (↔ eng [API-05] debug internals never in the client error envelope) | Are `display_errors`/`display_startup_errors` Off with `log_errors` On in prod? | S·r | Modern PHP Security |
| PHP-15 | SQL built with string concat of `$_GET`/`$_POST` (even via PDO `query`/`exec`) | **Bind parameters with PDO/ORM** — quote breakout becomes SQLi regardless of driver (↔ [WEB-01], eng [DATA-12] the same binding failure seen at the framework, web, and data layers) | Is every query using prepared statements/bound values? | B·w | Modern PHP Security |
| PHP-16 | Outbound HTTP/file URL from user input (`file_get_contents($url)`, curl) without a destination policy | **Constrain server-side fetches** — SSRF hits cloud metadata and internal FastCGI/admin (↔ [WEB-07] attacker-chosen fetch destinations reach link-local and admin nets) | Is the destination allowlisted or blocked from private/link-local ranges? | B·w | Modern PHP Security |

## 6. Secure Design Principles

> The cross-cutting engineering principles beneath every section above, from Ross Anderson, *Security Engineering*, 3rd ed. (`anderson-security-engineering`). Where a principle is the general form of a specific rule (least privilege → [SEC-04]/[PG-01]/[SEC-16]; complete mediation → [WEB-31]/[WEB-33]), the specific rule cross-links here rather than restating it. See [Principle 14].

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| SECD-01 | Any system carries a protection claim ("secure", "safe", "compliant") | **Write the security policy first** — state in one paragraph what must not happen and who may do what, before choosing mechanisms; mechanism without a falsifiable policy is theatre (↔ biz [PROD-09] write the channel contract before the components; ↔ a11y [A11Y-22] "secure" is a falsifiable sentence or it is empty; ↔ biz [CLM-05] three hostile readers who check every claim against a written spec; ↔ eng [RLSE-09] expert sign-off is exact quoted strings and yes/no questions, not jargon that shifts liability) | Can an engineer falsify compliance with the policy in a test? | S·p | anderson-security-engineering |
| SECD-02 | A component whose compromise would break the policy | **Minimize the trusted computing base** — treat it as **trusted** and shrink that set: split duties, sandbox untrusted code, remove unneeded keys and scopes, because a fat TCB makes every bug fatal (↔ [SEC-04] a jailbreak exercises every scope it holds; ↔ [PG-01] app SQLi is cluster takeover if the role is superuser; ↔ [SEC-16] one identity that can silently swap production weights; ↔ [PG-06] PUBLIC grants hand every role unintended DML; ↔ [PG-14] an over-privileged ops role expands the blast radius of a stolen credential; ↔ [Principle 14]) | If this principal is malicious, is the whole policy dead? | B·p | anderson-security-engineering |
| SECD-03 | Any security-relevant read / write / exec path | **Check every access at the reference monitor** — authorize at the reference-monitor boundary on *each* use, not only at session start (Saltzer's **complete mediation**), because check-once/use-many, CSRF, and TOCTTOU mint unearned authority (↔ [WEB-31], [WEB-33] complete mediation is deny-by-default authorization on every path) | Where is the check on *this* call, and what if the ACL changed since open? | B·w | anderson-security-engineering |
| SECD-04 | An action can create or destroy value, or erase audit evidence | **Dual control for high impact** — require two independent authorizers or mechanism classes, because a single insider or single malware path is the historical failure mode (↔ [SEC-05] semantic human gate) | Can one person with one laptop finish this irreversibly? | B·p | anderson-security-engineering |
| SECD-05 | Installer defaults, cloud templates, API failure paths | **Fail-safe defaults** — default deny and degrade to a documented safe state rather than open privilege or network, because the default is the configuration most deployments keep (↔ [WEB-33] deny-by-default is the only safe installer and failure posture) | What does a naive deploy expose in five minutes? | B·w | anderson-security-engineering |
| SECD-06 | A serious hazard rests on a single choke-point control | **Independent defense in depth** — require ≥2 mechanisms that fail *independently* (correlated clones don't count), because single-layer bypass is routine and fault masking hides an eroded margin | If the first layer is fully owned, what still stops the loss? | S·p | anderson-security-engineering |
| SECD-07 | A new product, major redesign, or "secure AI feature" kickoff | **Threat-model before mechanism** — list assets, attacker assets, and threat-tree / STRIDE paths (including insiders and supply chain) before buying controls, because malice forces the least-convenient failure (↔ biz [STRAT-02] name the ways the plan dies before the forward plan; ↔ biz [STRAT-14] name the worst cases and repair paths before a reversible decision freezes) | What are the top five ways this policy dies, including insiders? | S·p | anderson-security-engineering |
| SECD-08 | Humans must uphold the policy (consent, install, on-call emergency UI) | **Make the safe path the easy path** — design so the least-effort and high-stress actions are the safe ones (**psychological acceptability**), because dialog fatigue and baroque controls train bypass (↔ interaction [COG-01]; human gates must stay usable under load, [SEC-05]) | Under time pressure, what will a competent user actually do? | S·p | anderson-security-engineering |
| SECD-09 | The incident-response plan for a template or gallery compromise lists re-enrollment, reset, or credential rotation as the remediation | **An unconcealed identifier cannot be reissued** — when a match path trusts stored body-derived identifiers (face, fingerprint, iris, voice), the compromise runbook must name a remediation the subject can actually execute: revoke the stored templates and drop that factor from the accept path until supervised re-capture, because the subject cannot be issued a new face, so "reset the credential" is a step no one can perform and the leaked template remains a valid presentation against every system holding the same person (↔ [SECD-05] the failure state must be a documented safe one) | After a gallery leak, which remediation step does not require the subject to change their body? | B·p | face-recognition-compulsory-visibility |
| SECD-10 | A state or authentication diagram shows biometric enrollment as the only route to a function, with no second accept path | **A biometric factor is not a sole path** — when enrollment of a body-derived identifier is required to reach a function, ship a second path of equal function (credential, token, staffed override), or record in the policy document the statutory basis and the scope it is limited to, because a matcher's false-non-match rate is not uniform across cohorts (↔ ml [FAIR-03], [FAIR-04] name the per-cohort error at the production threshold; this row is what that error becomes when the path is sole — an exclusion rather than a retry) so a sole biometric path converts a per-cohort error rate into a per-cohort exclusion with no recovery, and subjects who cannot enroll at all — injury, capture failure, refusal — have no route (↔ [SECD-08] the safe path must also be a reachable one) | Can a user finish this task without enrolling a biometric, and by what path? | S·p | face-recognition-compulsory-visibility |

## 7. Cross-lexicon links

Security is a spanning concern, so many rows restate a rule another lexicon owns from the security angle:

- **Injection is one mechanism across three sinks.** SQL parameterization [WEB-01]/[PHP-15]/[WP-01]/[PG-07] is engineering's [DATA-12] read as an attack; the LLM analogue is [SEC-02] (treat model output as untrusted before any interpreter).
- **Least privilege repeats up the stack.** [PG-01] (non-superuser app role) and [SEC-04] (least-privilege agency) are the same rule at the database and the model; both bound the blast radius of a compromised component (↔ eng [ARCH-02], single writer owns the table).
- **Secrets discipline is one rule.** [WEB-16]/[PG-09] and [SEC-06] all say the same thing: anything a component can read, an attacker who reaches that component can read too.
- **Rate limiting is denial-of-wallet and DoS at once.** [WEB-17] and [SEC-08] share a source mechanism; unmetered access is both an availability and a cost failure.
- **Validation at the trust boundary** ([SEC-01], [PHP-08]) is the security face of the same boundary the engineering lexicon guards for correctness ([API-11] round-trip, [DATA-03] schema compat).
