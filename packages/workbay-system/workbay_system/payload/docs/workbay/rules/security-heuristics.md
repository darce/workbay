# Security Heuristics Lexicon

## About this document

> **Referenced, not read end-to-end.** A dedicated rubric beside engineering, business/marketing, design/aesthetics, writing, and accessibility. It governs **security as a spanning concern**: the trust boundaries, injection sinks, privilege grants, and misconfigurations that turn a working feature into an exploitable one. Like accessibility, security cuts across the other lexicons — an SQL-injection rule is also a data rule, a least-privilege rule is also an architecture rule — so it earns its own lexicon rather than scattering.

> **Provenance caveat — this is a bootstrap pass.** Only two sections are grounded in a distilled source today: **§1 (LLM & agent security)** is sourced to Steve Wilson, *The Developer's Playbook for Large Language Model Security* (`llm-security-playbook`), and several **§3 (PHP)** rows to *Modern PHP Security* (Chauchefoin & David, sec4dev 2020). Every row whose `Src` reads **`bootstrap`** is a first-pass rule with **no distilled source yet** — drawn from established web/PHP/WordPress/PostgreSQL security practice but not yet traced to a text in the corpus. A later pass will re-ground these against canonical sources (the library currently lacks the security-engineering canon — Anderson, Shostack, Stuttard, Aumasson), and unsourced rows are held at tier **S/J** accordingly. Treat `bootstrap` rows as strong defaults under review, not settled Blockers-by-citation.

**Scope.** Deliberately narrow for now: LLM/agent security, general web application security, and the PHP / WordPress / PostgreSQL stack. It does **not** yet cover cryptography, network security, OS hardening, or identity protocols in depth — those await both the sources and the pass.

Row format matches the sibling lexicons: `ID | Trigger | Rule | Answers | T·P | Src` — tier **B**locker (exploitable, data-loss, or auth-bypass if violated) / **S**hould (strong default) / **J**udgment; phase **p**lan (architecture/threat-model) / **w**rite (code) / **r**eview (config/audit). `Src` names the distilled source, or `bootstrap` for a not-yet-sourced row. IDs are stable citation keys (`[WEB-01]`) and immutable once assigned. Cross-lexicon borders cite `↔ eng/biz` rules instead of restating them.

## 1. LLM & Agent Security

> Moved here from the engineering lexicon (`SEC-01..10` keep their IDs; existing `[SEC-02]` citations still resolve). Source: Wilson, *The Developer's Playbook for Large Language Model Security*.

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| SEC-01 | Data crossing any component boundary (user↔app, app↔LLM, LLM↔services) | **Validate at every trust boundary** — trust changes at each crossing; the LLM's output side is a boundary too | Where does trust change, and what validation runs there — in and out? | S·p | llm-security-playbook ch-3 |
| SEC-02 | LLM output passed to SQL, shell, eval, or rendered as HTML | **Treat LLM output as untrusted input** — generated text can carry injected payloads into any interpreter (↔ [WEB-01], [WEB-04]) | Is the output parameterized, escaped, or encoded before the sink? | B·r | llm-security-playbook ch-7 |
| SEC-03 | User or external content interpolated into an LLM prompt | **Delimit untrusted prompt segments** — the model can't distinguish data from instructions unless the boundary is marked | Is every untrusted segment structurally marked and out of instruction position? | S·w | llm-security-playbook ch-4 |
| SEC-04 | LLM granted a tool, plugin, API scope, or DB permission | **Least-privilege agency** — a jailbroken model exercises every permission it holds (↔ [PG-01]) | Is this the minimum scope the feature needs? | B·p | llm-security-playbook ch-7 |
| SEC-05 | LLM can trigger an irreversible, financial, or destructive action | **Human-in-the-loop gate** — autonomy plus prompt injection equals attacker-driven transactions | Does a human approve before the side effect executes? | B·p | llm-security-playbook ch-7 |
| SEC-06 | Secrets or PII in a prompt template, fine-tune set, or RAG store | **Don't teach the model what it must never say** — anything the LLM can read can be extracted by any user (↔ [WEB-16]) | What happens if this datum is disclosed verbatim? | B·p | llm-security-playbook ch-5 |
| SEC-07 | Web pages, files, or DB rows fed into a RAG prompt | **Retrieved content is attacker content** — indirect injection rides in on documents | Is retrieved text screened and confined to a data role? | S·w | llm-security-playbook ch-4 |
| SEC-08 | Public LLM endpoint or feature without usage limits | **Rate-limit and cap per caller** — unmetered access invites DoS and denial-of-wallet (↔ [WEB-17]) | Are per-caller limits, token caps, and budget alerts configured? | S·w | llm-security-playbook ch-8 |
| SEC-09 | LLM-suggested package added to a dependency manifest | **Verify hallucinated dependencies** — attackers register malware under names models invent | Does this package exist with the expected owner and real history? | B·r | llm-security-playbook ch-6 |
| SEC-10 | Model or dataset pulled from a hub at build/run time | **Pin and record provenance** — unpinned weights are an unauditable dependency | Is the exact revision pinned and integrity verified? | S·w | llm-security-playbook ch-9 |

## 2. Web Application Security

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| WEB-01 | User-controlled data concatenated into an SQL string (or interpolated into a query without a binder) | **Parameterize every query** — string-built SQL lets any quote/terminator break out into code; use prepared statements + bound params (↔ eng [DATA-12]) | Is every SQL value bound, never concatenated? | B·w | bootstrap |
| WEB-02 | User input passed to `exec`/`system`/`shell_exec`/`Runtime.exec`/`ProcessBuilder` without an argv array | **Never shell-interpolate untrusted input** — metacharacters become a second command; use argv APIs or refuse shell | Is the OS command built as discrete args with no shell? | B·w | bootstrap |
| WEB-03 | LDAP filter built with string concat of request fields | **Escape or bind LDAP filters** — `)(uid=*` style input widens the filter and bypasses authz | Is every LDAP filter value escaped per RFC 4515? | B·w | bootstrap |
| WEB-04 | Response body echoes request data (or stored user content) into HTML without context-aware encoding | **Encode for the sink context** — unescaped HTML lets scripts run as the victim (reflected/stored XSS) | Is every untrusted string escaped for HTML/attr/JS/URL before emit? | B·w | bootstrap |
| WEB-05 | Client-side code writes `location`/`document.write`/`innerHTML` from `location.hash`/`postMessage`/URL params | **Treat DOM sinks as XSS sinks** — client-only flows still execute attacker script in the origin | Is untrusted data never assigned to HTML/JS sinks without sanitization? | B·w | bootstrap |
| WEB-06 | State-changing POST/PUT/DELETE lacks a per-session CSRF token verified server-side (`SameSite` cookies are defense-in-depth, not a substitute) | **Require anti-CSRF on mutations** — a third-party site can forge the browser's cookies into a state change | Does every state change verify a CSRF token or equivalent origin binding? | B·w | bootstrap |
| WEB-07 | Server fetches a URL/host derived from user input (webhooks, previews, importers) | **Block SSRF to internal nets** — attacker-chosen URLs reach link-local/metadata/admin services; allowlist destinations or deny private ranges after re-resolve | Is the fetch target allowlisted or proven non-internal after DNS-rebinding-safe checks? | B·w | bootstrap |
| WEB-08 | Object ID / path param used in a query without comparing ownership or tenant to the authenticated principal | **Authorize the object, not just the route** — knowing another ID is enough for IDOR/data disclosure | Does the handler enforce ownership/tenant before read or write? | B·w | bootstrap |
| WEB-09 | Session ID accepted from URL, or session not regenerated on privilege change (login) | **Rotate session ID on auth** — fixation reuses a pre-set SID after login as the victim | Is a new session ID issued at login and the SID never taken from the URL? | B·w | bootstrap |
| WEB-10 | Auth cookie set without `Secure`+`HttpOnly` (+ appropriate `SameSite`) on a TLS site | **Lock down cookie flags** — missing flags enable theft via XSS or leakage over HTTP | Are session cookies Secure, HttpOnly, and SameSite-scoped? | S·r | bootstrap |
| WEB-11 | Password login for privileged accounts with no second factor or step-up | **Require MFA for privileged auth** — password-only admin paths fall to phishing and reuse | Is MFA (or step-up) required for admin and high-risk actions? | S·p | bootstrap |
| WEB-12 | Upload accepts arbitrary extension/content-type and stores under a web-served path | **Store uploads outside exec root** — executable or HTML uploads become RCE or stored XSS | Is the upload non-executable, type-validated, and outside the docroot? | B·w | bootstrap |
| WEB-13 | User path segment joined to a base directory without canonicalization + prefix check | **Reject path traversal** — `../` escapes the intended root and reads secrets | Is the resolved path verified to stay under the allowed base? | B·w | bootstrap |
| WEB-14 | Redirect target (`?next=`, `return_url`) taken from the request without a same-origin/allowlist check | **Allowlist redirect targets** — off-site redirects turn the app into a phishing trampoline | Is the redirect target same-site or on an explicit allowlist? | S·w | bootstrap |
| WEB-15 | App responses omit CSP and HSTS (or set weak/default headers) on HTTPS endpoints | **Ship CSP + HSTS** — missing headers leave XSS impact high and enable SSL strip | Are CSP and HSTS present and non-permissive on all HTTPS responses? | S·r | bootstrap |
| WEB-16 | API keys, passwords, or private keys appear in source, committed config, or client bundles | **Never commit secrets** — repo/history exposure is permanent credential theft (↔ [SEC-06]) | Are secrets only in a secret store or env, never in VCS or client code? | B·r | bootstrap |
| WEB-17 | Auth, password-reset, or expensive endpoints have no per-IP/account rate limit | **Rate-limit abuse endpoints** — unbounded attempts enable credential stuffing and DoS (↔ [SEC-08]) | Are login/reset/OTP routes rate-limited and lockout-aware? | S·w | bootstrap |
| WEB-18 | TLS terminated with outdated protocols/ciphers, or mixed-content/HTTP admin allowed | **Enforce modern TLS only** — weak TLS or cleartext admin enables MitM of credentials | Is TLS 1.2+ only, with no HTTP admin and HSTS preload-ready? | B·r | bootstrap |
| WEB-19 | Framework mass-assigns request body fields onto a model/entity (including `role`/`is_admin`) | **Allowlist assignable fields** — extra JSON keys escalate privilege via mass assignment (↔ eng [API-11]) | Is the binder restricted to an explicit safe field list? | B·w | bootstrap |
| WEB-20 | `unserialize`/`pickle`/`ObjectInputStream`/binary object decode on untrusted bytes | **Refuse untrusted deserialization** — gadget chains turn bytes into RCE (↔ [PHP-02]) | Is untrusted input never deserialized with a native object format? | B·w | bootstrap |

## 3. PHP Security

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| PHP-01 | Authz/authn or token compare uses `==`/`!=` between request data and secrets/IDs | **Use strict comparison** — type juggling (`0e…` magic hashes, `"0"==false`) bypasses equality checks; prefer `===`/`hash_equals` | Are all security-sensitive compares strict (`===`/`hash_equals`)? | B·w | bootstrap |
| PHP-02 | `unserialize()` called on cookies, POST bodies, cache blobs, or queue payloads | **Never unserialize untrusted data** — POP gadget chains in app/framework classes become RCE (↔ [WEB-20]) | Is every `unserialize` removed or limited to signed input + `allowed_classes`? | B·w | Modern PHP Security |
| PHP-03 | Filesystem API (`file_exists`, `file_get_contents`, `getimagesize`, …) takes a user-influenced path | **Block phar:// side-effects** — path wrappers deserialize PHAR metadata and run object chains without calling `unserialize` | Are user paths blocked from `phar://` and other dangerous wrappers? | B·w | Modern PHP Security |
| PHP-04 | `include`/`require` path built from request params or remote URLs | **Pin include roots** — user-controlled includes are LFI/RFI and code execution | Is the include path from a fixed map, never raw user input? | B·w | bootstrap |
| PHP-05 | `extract($_GET)` / `extract($_POST)` / `parse_str` into the current symbol table without `EXTR_SKIP` | **Do not extract request into scope** — attackers overwrite existing variables (register_globals class) | Is `extract` on request data absent or forced to non-overwrite flags? | B·w | bootstrap |
| PHP-06 | `preg_replace` with the `/e` modifier (or a dynamic pattern from user input) | **Ban preg_replace /e** — the replacement is evaluated as PHP code | Is `/e` absent and are patterns never user-controlled? | B·w | bootstrap |
| PHP-07 | Production `php.ini` leaves `exec`/`system`/`passthru`/`shell_exec`/`proc_open` enabled without need | **Disable unused dangerous functions** — post-RCE or LFI impact collapses if dangerous symbols are removed | Is `disable_functions` set and verified against actual app needs? | S·r | Modern PHP Security |
| PHP-08 | Superglobals used raw for business decisions without `filter_input`/`filter_var` or allowlist validation | **Validate at the boundary** — untyped input reaches sinks; filter or schema-validate before use (↔ [SEC-01]) | Is every external input validated to type/range before logic or sinks? | S·w | bootstrap |
| PHP-09 | Passwords stored or checked with `md5`/`sha1`/`crypt` without a modern KDF | **Use password_hash/password_verify** — fast hashes are offline-crackable; `password_hash` uses adaptive algorithms | Are all password stores using `password_hash` (argon2/bcrypt)? | B·w | bootstrap |
| PHP-10 | App can read paths outside its tree; `open_basedir` unset in shared/hosting PHP | **Constrain open_basedir** — LFI and path bugs cannot escape the allowed tree | Is `open_basedir` set to app + upload dirs only in prod? | S·r | Modern PHP Security |
| PHP-11 | Echo/`<?=` of request or DB strings into HTML without `htmlspecialchars(..., ENT_QUOTES, 'UTF-8')` or template auto-escape | **Escape every HTML emit** — PHP defaults do not encode output; XSS follows (↔ [WEB-04]) | Is all HTML output escaped (or auto-escaped by the template engine)? | B·w | bootstrap |
| PHP-12 | Upload handler trusts `$_FILES['type']` or the client filename extension alone | **Verify content, not client MIME** — spoofed types upload PHP polyglots; check extension allowlist + content sniff + no-exec storage | Is the extension allowlisted and the file stored non-executable outside docroot? | B·w | Modern PHP Security |
| PHP-13 | Class name for `new $class(...)` taken from routing/request without an allowlist | **Allowlist dynamic instantiation** — attacker-chosen classes run dangerous constructors | Is the instantiable class set a fixed allowlist? | B·w | Modern PHP Security |
| PHP-14 | Prod config has `display_errors=On` or broad `error_reporting` printed to clients | **Hide errors from clients** — stack traces leak paths, queries, and secrets; log instead | Are `display_errors`/`display_startup_errors` Off with `log_errors` On in prod? | S·r | Modern PHP Security |
| PHP-15 | SQL built with string concat of `$_GET`/`$_POST` (even via PDO `query`/`exec`) | **Bind parameters with PDO/ORM** — quote breakout becomes SQLi regardless of driver (↔ [WEB-01], eng [DATA-12]) | Is every query using prepared statements/bound values? | B·w | Modern PHP Security |
| PHP-16 | Outbound HTTP/file URL from user input (`file_get_contents($url)`, curl) without a destination policy | **Constrain server-side fetches** — SSRF hits cloud metadata and internal FastCGI/admin (↔ [WEB-07]) | Is the destination allowlisted or blocked from private/link-local ranges? | B·w | Modern PHP Security |

## 4. WordPress Security

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| WP-01 | `$wpdb->query`/`get_results`/`get_var` with interpolated variables and no `$wpdb->prepare` | **Prepare every $wpdb query** — unprepared SQL is classic SQLi in plugins/themes (↔ [WEB-01]) | Is every variable SQL path using `$wpdb->prepare`? | B·w | bootstrap |
| WP-02 | Form/admin/AJAX handler that changes state without `wp_verify_nonce` / `check_admin_referer` | **Verify nonces on mutations** — missing nonces enable CSRF against logged-in users (↔ [WEB-06]) | Does every state-changing handler verify a nonce? | B·w | bootstrap |
| WP-03 | `add_menu_page`/`add_submenu_page` or a sensitive action runs without `current_user_can(...)` | **Gate admin with capabilities** — UI visibility is not authorization; a missing cap is privilege abuse (↔ [WEB-08]) | Is `current_user_can` checked for the minimum required capability? | B·w | bootstrap |
| WP-04 | Request data saved or used without the `sanitize_*` / `wp_kses` appropriate to the field | **Sanitize on input** — raw POST stored as options/postmeta becomes stored XSS or logic bugs later | Is every persisted field sanitized with the right `sanitize_*` helper? | S·w | bootstrap |
| WP-05 | Theme/plugin echoes post meta, options, or request data without `esc_html`/`esc_attr`/`esc_url`/`wp_kses_post` | **Escape on output** — WP does not auto-escape; a context-wrong `esc_*` is still XSS (↔ [WEB-04]) | Is every echo using the correct `esc_*` for its HTML context? | B·w | bootstrap |
| WP-06 | `register_rest_route` omits `permission_callback` or uses `__return_true` on privileged routes | **Set REST permission_callback** — default-open routes expose data and mutations anonymously | Does every non-public route implement a real `permission_callback`? | B·w | bootstrap |
| WP-07 | XML-RPC enabled on a site that does not need remote publishing/pingbacks | **Disable unused XML-RPC** — `xmlrpc.php` enables brute-force amplification and pingback SSRF | Is XML-RPC off (or tightly restricted) if unused? | S·r | bootstrap |
| WP-08 | Plugins/themes not updated; abandoned extensions with known CVEs still active | **Update or remove abandoned code** — unpatched plugins are the dominant WP compromise path | Are all active plugins/themes maintained and currently patched? | B·r | bootstrap |
| WP-09 | `DISALLOW_FILE_EDIT` not defined true in `wp-config.php` | **Disable the file editor** — dashboard editors turn XSS or admin access into persistent RCE | Is `DISALLOW_FILE_EDIT` true in production config? | S·r | bootstrap |
| WP-10 | A `wp_ajax_nopriv_*` hook performs privileged work or returns sensitive data | **Treat nopriv as public** — `nopriv` handlers are anonymous attack surface | Does every `nopriv` handler expose only intentional public behavior? | B·w | bootstrap |
| WP-11 | Custom upload or `wp_handle_upload` path allows executable extensions or serves from an executable location | **Lock down media uploads** — arbitrary PHP/HTML in uploads becomes webshell or stored XSS (↔ [WEB-12]) | Are allowed MIME/extensions strict and uploads non-executable? | B·w | bootstrap |
| WP-12 | `AUTH_KEY`/`SECURE_AUTH_KEY`/`LOGGED_IN_KEY`/`NONCE_KEY` (and salts) left as defaults or duplicated | **Unique strong salts** — default or reused keys weaken cookie auth and nonce strength | Are all eight keys/salts unique, long, and not from public samples? | B·r | bootstrap |
| WP-13 | Plugin/theme PHP files load application code without a `defined('ABSPATH') \|\| exit;` guard | **Guard direct file access** — direct hits bypass WP bootstrap and leak or execute partial includes | Do plugin/theme PHP entry files refuse direct access via an ABSPATH check? | S·w | bootstrap |
| WP-14 | Options update / user-meta write accepts `role`, `capabilities`, or serialized objects from the request | **Block privilege mass-assignment** — user-controlled role fields create admin accounts (↔ [WEB-19]) | Are role/capability fields excluded from user-editable forms and APIs? | B·w | bootstrap |

## 5. PostgreSQL Security

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| PG-01 | Application DB role is superuser or holds cluster-wide powerful attributes (`rds_superuser`, `BYPASSRLS`) | **Run the app as least privilege** — app SQLi becomes cluster takeover if the role is superuser (↔ [SEC-04], eng [ARCH-02]) | Is the app role non-superuser with only the schema privileges it needs? | B·r | bootstrap |
| PG-02 | Multi-tenant tables rely only on app `WHERE` clauses; RLS disabled or no policies | **Enable RLS policies** — a missed filter leaks rows; RLS enforces tenant isolation in the DB | Is RLS ON with policies that match the tenancy model? | B·p | bootstrap |
| PG-03 | `SECURITY DEFINER` function without a pinned `search_path` (or a writable schema ahead in the path) | **Pin search_path on DEFINER** — attackers plant shadow objects and hijack privileged functions | Do DEFINER functions set a fixed safe `search_path` and a trusted owner? | B·w | bootstrap |
| PG-04 | `pg_hba.conf` uses `trust`/`md5` for remote or broad CIDR matches | **Prefer scram-sha-256 auth** — `trust` is passwordless access; md5 is weaker than SCRAM | Are remote entries scram-sha-256 (not trust/md5) with tight CIDRs? | B·r | bootstrap |
| PG-05 | Connections allowed without SSL, or `sslmode` defaults to prefer/disable in app DSNs | **Require TLS to Postgres** — cleartext credentials and rows are MitM-visible | Is server SSL on and are clients using verify-full/verify-ca as appropriate? | B·r | bootstrap |
| PG-06 | `GRANT … TO PUBLIC`, or default public-schema privileges left world-usable | **Revoke PUBLIC grants** — PUBLIC privileges give every role unintended DML/EXECUTE | Are unnecessary PUBLIC grants revoked on schemas, tables, and functions? | S·r | bootstrap |
| PG-07 | PL/pgSQL or SQL builds dynamic statements with string concat of identifiers/literals | **Use format(%I/%L) or EXECUTE … USING** — dynamic SQL at the DB layer is still SQLi (↔ [WEB-01]) | Is every dynamic identifier/literal passed via `%I`/`%L` or a bind param? | B·w | bootstrap |
| PG-08 | Untrusted users can `CREATE EXTENSION`, or extensions installed cluster-wide without review | **Treat extensions as code** — unpackaged C/SQL extensions expand attack surface and privilege | Are only required extensions installed, by a superuser, after review? | S·r | bootstrap |
| PG-09 | DB passwords in app repos, a world-readable `.pgpass`, or a shared superuser password across envs | **Isolate DB secrets** — a leaked password equals full data access (↔ [WEB-16]) | Are DB credentials unique per env and stored only in a secret manager? | B·r | bootstrap |
| PG-10 | No base backups + WAL archiving, or an untested PITR restore path | **Prove backups restore** — untested backups fail when ransomware or a `DROP` happens (↔ eng [RES-07]) | Is PITR configured and a restore drill documented as successful? | B·p | bootstrap |
| PG-11 | `log_statement`/`pgaudit` off for privileged and DDL activity; no log shipping | **Log privileged SQL** — without an audit trail, abuse and SQLi exfiltration go unnoticed | Are DDL/role changes and sensitive statements logged and retained? | S·r | bootstrap |
| PG-12 | `max_connections` high with no per-role/`pgbouncer` limits; app opens unbounded pools | **Cap connections** — connection storms exhaust the postmaster and cause an outage (↔ eng [RES-14]) | Are server and pooler limits set below collapse thresholds per role? | S·r | bootstrap |
| PG-13 | Views/functions expose rows that RLS on base tables would hide (`security_barrier`/owner ignored) | **Align views with RLS** — a non-barrier view can leak filtered rows | Do views and DEFINER paths preserve the intended row filters? | S·w | bootstrap |
| PG-14 | Replication or backup role holds broad write privileges beyond need | **Separate backup/repl roles** — an over-privileged ops role expands blast radius if stolen | Do backup/replication roles have only the minimal REPLICATION/read grants? | S·r | bootstrap |

## 6. Cross-lexicon links

Security is a spanning concern, so many rows restate a rule another lexicon owns from the security angle:

- **Injection is one mechanism across three sinks.** SQL parameterization [WEB-01]/[PHP-15]/[WP-01]/[PG-07] is engineering's [DATA-12] read as an attack; the LLM analogue is [SEC-02] (treat model output as untrusted before any interpreter).
- **Least privilege repeats up the stack.** [PG-01] (non-superuser app role) and [SEC-04] (least-privilege agency) are the same rule at the database and the model; both bound the blast radius of a compromised component (↔ eng [ARCH-02], single writer owns the table).
- **Secrets discipline is one rule.** [WEB-16]/[PG-09] and [SEC-06] all say the same thing: anything a component can read, an attacker who reaches that component can read too.
- **Rate limiting is denial-of-wallet and DoS at once.** [WEB-17] and [SEC-08] share a source mechanism; unmetered access is both an availability and a cost failure.
- **Validation at the trust boundary** ([SEC-01], [PHP-08]) is the security face of the same boundary the engineering lexicon guards for correctness ([API-11] round-trip, [DATA-03] schema compat).

## Consumption

Canonical source. Consuming repos sync this file and cite rules by ID (`[WEB-01]`, `[SEC-04]`). The `§1` LLM rules moved here from engineering keep their `SEC-*` IDs. Because §§2–5 are a bootstrap pass, a consumer wiring `bootstrap` rows into a gate should treat them as strong review defaults, not cited Blockers, until the grounding pass lands. Product-specific security targets (which controls, which environments, by when) live in the consuming project, not here.
