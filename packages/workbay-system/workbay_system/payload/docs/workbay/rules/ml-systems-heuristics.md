# Machine-Learning Systems Heuristics Lexicon

## About this document

> **Referenced, not read end-to-end.** A decision rubric beside engineering, graph-theory, security, and interaction-ux. It governs one thing: **the evidence, uncertainty, and operations of a deployed model** — how vectors are made, compared, calibrated, versioned, exposed, and corrected. It begins where the graph lexicon's construction rules end: `[GRPH-17..23]` build and cut a similarity graph; this lexicon governs the embeddings that feed it, the thresholds that accept its edges, the cohorts it is measured on, and the provenance that lets a claim be reproduced.

**What it covers.** Fourteen populated families. **MLDATA** (dataset construction, splits, leakage, representativeness) · **EMB** (embedding spaces, normalization, similarity, versioning) · **EVAL** (offline / pipeline / user-level evaluation) · **CAL** (confidence, thresholds, calibration, abstention) · **DRIFT** (data / concept / model / threshold drift) · **SERVE** (inference, action safety, serving-path architecture) · **FAIR** (cohort and capture-condition performance) · **PROV** (model / dataset / evidence / derived-output provenance) · **HITL** (human review, correction, active learning, adjudication) · **TRACK** (temporal association and identity continuity across video frames — feature/object tracking, drift, occlusion, data association) · **VSEG** (temporal structure of a media timeline — shot-boundary detection, gradual transitions, shot→scene grouping, keyframe selection; where TRACK follows one entity *across* frames, VSEG cuts the timeline *into* units) · **COST** (compute budgets and cost per accepted output — SLOs, serving cost, capacity, retraining economics) · **FM** (foundation-model application composition and adaptation — instruction channels, prompt/context construction, structured-output contracts, the adaptation ladder, agent-loop topology; how a pretrained generative model is configured and composed, not every rule that mentions an LLM) · **RAG** (retrieval-augmented generation pipelines — chunking, retrieval/generation separation, reranking, context selection, abstention, citation, index freshness, deletion propagation). TRACK is the temporal layer above EMB (appearance) and CAL (match threshold); COST is the budget layer beside SERVE, and it cross-links engineering `PERF-*`/`OBS-*` for generic performance rather than restating them. New rows enter at the lowest defensible tier and are promoted on independent convergence.

> **Provenance — primary-source, section-traced.** Unlike the security bootstrap, every row here cites a primary source distilled to `distilled/ml-systems/` with section/figure anchors: Huyen, *Designing Machine Learning Systems* (O'Reilly 2022); Mitchell et al., *Model Cards for Model Reporting* (ACM FAT\* 2019); Buolamwini & Gebru, *Gender Shades* (PMLR 81, 2018); Grother, Ngan & Hanaoka, *FRVT/FRTE Part 3: Demographic Effects* (NIST IR 8280, 2019 — now under NIST's FRTE program, complemented by IR 8429 (2022) and the continuously-updated FRTE demographic tables); Kim, Jain & Liu, *AdaFace* (CVPR 2022); Sculley et al., *Hidden Technical Debt in Machine Learning Systems* (NeurIPS 2015); Monarch, *Human-in-the-Loop Machine Learning* (Manning 2021); Huyen, *AI Engineering: Building Applications with Foundation Models* (O'Reilly 2025); Lakshmanan, Robinson & Munn, *Machine Learning Design Patterns* (O'Reilly 2020). The `Src` column names the distillation slug; the depth, exemptions, and worked failures live there.

**Scope boundary.** Decision heuristics for model evidence and operations, not an ML textbook: *what a model result must carry and be measured against*, not how to train a network. Deep algorithm mechanics stay in the sources.

Row format matches the sibling lexicons: `ID | Trigger | Rule | Answers | T·P | Src` — tier **B**locker (a wrong answer corrupts correctness, ships real-world harm, or breaks security) / **S**hould (strong default) / **J**udgment; phase **p**lan (design / data / eval design) / **w**rite (implementation) / **r**eview (ship gate). IDs (`[FAIR-01]`) are stable citation keys, immutable once assigned. Cross-lexicon borders cite `↔ grph/sec/eng [X]` rather than restating them.

## 1. MLDATA — Dataset construction, splits, representativeness

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| MLDATA-01 | A convenience scrape or public dump is the sole training corpus; no sampling frame named | **Reject bare nonprobability sampling** — name the sampling method and add probability/**stratified** coverage of the production strata; a convenience-only corpus ships a selection-biased model that fails off-slice | Is the sampling method named, and are production strata covered beyond convenience? | S·p | designing-ml-systems |
| MLDATA-02 | Global random split; a rare but required class sits below ε in the population | **Stratify the rare strata** — sample per stratum with minimum counts; a simple random split omits rare classes and the model behaves as if they do not exist | Do post-sample histograms still contain every class we must detect? | S·w | designing-ml-systems |
| MLDATA-03 | Multi-annotator or multi-vendor labels merged into one "gold" set silently | **Resolve label multiplicity in writing** — require a written disagreement rule plus annotator training, and keep the pre-adjudication labels; otherwise "gold" is ambiguous and models diverge | What is the written rule when annotators disagree? | S·w | designing-ml-systems |
| MLDATA-04 | A new labeled batch is merged with no per-sample source / annotator / batch fields | **Persist data lineage** — keep the origin of each sample *and* each label; without it a poisoned batch is irreversible and debugging fails (↔ grph [GRPH-14] both fail when an artifact cannot name the input that produced it) | Can we filter train/eval by label source and acquisition batch? | B·w | designing-ml-systems |
| MLDATA-05 | An implicit negative is recorded after a fixed click/interaction window (CTR, fraud) | **Window the delayed feedback** — document the natural-label **feedback-loop length** and measure post-window positives before trusting the negative cutoff | What fraction of positives arrive after our negative cutoff? | S·p | designing-ml-systems |
| MLDATA-06 | A vision fairness audit labels people only by race / ethnicity / country | **Representative skin-tone labels for appearance tasks** — label with a skin-tone scale validated for representative coverage of the deployment population (e.g. the 10-point **Monk Skin Tone** scale), not a UV-response scale that collapses darker tones (Fitzpatrick I–VI); race is unstable across geography and hides within-category skin-tone range | Are representative skin-tone labels present — a tone scale, not race proxies or a UV scale? | S·w | gender-shades |
| MLDATA-07 | Benchmark composition shows one intersection cell ≪ the others while a global claim is made | **Intersectional balance before a global claim** — report unique-subject counts per intersection; if severely imbalanced versus deployment, build or adopt a balanced holdout (the PPB pattern) and re-measure (↔ biz [AIPX-04] an aggregate claim over a hiding cell is the same global-clear failure at ship review) | What percentage of unique subjects falls in each intersection cell? | S·p | gender-shades |
| MLDATA-08 | A fairness-sensitive dataset is built solely by running a face detector over web images | **Detector-harvested sets need a post-hoc stratum audit** — measure the resulting demographic/phenotype mix and supplement the missing cells; the detector's blind spots otherwise become the benchmark's | Did the detector's blind spots become the benchmark's? | S·w | gender-shades |
| MLDATA-09 | A quality-control or outlier filter removes the hard cases — extreme pose, occlusion, low resolution — from a corpus whose claim covers those conditions | **A filter that removes the regime under test deletes the test** — when cleaning a corpus, check each exclusion rule against the conditions the claim covers and keep the hard cells as declared strata rather than contaminants, because the filter's blind spot silently becomes the benchmark's: the number goes green by deleting the cases that would have moved it, and nothing in the report shows the deletion (↔ [MLDATA-08] the same hole when the collecting tool never captured the cell; eng [PERF-03] coordinated omission is this exact mechanism in a load generator) | Which exclusion rules ran, and does any of them remove a condition the claim covers? | S·p | janus-benchmark-c |
| MLDATA-10 | Training inputs are clean, high-quality captures while deployment inputs are degraded (blur, noise, compression, small scale), and no measurement of the target degradation exists | **Match training degradation to measured target statistics** — when labelled data from the deployment capture domain is scarce, measure the target's degradation statistics (sharpness, noise, compression, subject size) and synthesize that measured distribution over the clean training data, then re-measure on true target-domain data, because degradations chosen by eye close a gap nobody sized: a clean-trained model loses most of its accuracy exactly where the product runs, and a near-domain proxy that still contains easy captures hides it (↔ [EVAL-06] the evaluation-side twin; [FAIR-05] report the capture strata this produces) | Which measured target statistic sets the strength of each training degradation? | S·w | video-to-video-face-surveillance |
| MLDATA-11 | A large-margin or otherwise confidence-maximizing classification objective trains on automatically collected labels with no noise-handling path | **A margin objective needs a label-noise path** — when the objective pushes every sample past a margin, require an explicit noise treatment (cleaning, per-sample reweighting, multiple centres per class, or sampling) and audit the highest-loss and highest-confidence disagreements, because a margin objective spends capacity dragging mislabelled samples across the boundary and corrupts the class it drags them into — and the worst offender is a sharp, well-captured image with the wrong label, which every quality gate passes (↔ [EMB-04] a quality proxy cannot catch a clean image with a wrong label; [MLDATA-03] governs human-annotated labels, this governs harvested ones) | What detects a mislabelled but high-quality sample before the margin objective fits it? | S·p | handbook-face-recognition |
| MLDATA-12 | A feature reaches training or serving with no machine-checkable range, type, required-ness, or distribution expectation | **Feature expectations belong in a versioned schema** — write each feature's permitted values, types, required-ness, and expected distribution into a versioned schema, validate every training and serving batch against it, and alert when live data diverges, because model behaviour is specified by its data and a corrupted feature is invisible to a metric computed over the same corrupted data (record-level schema discipline is eng [MODEL-03]; poison-specific gates are sec [SEC-11]) | Which schema assertions ran on this batch, and what fires when live data leaves the expected range? | S·w | ml-test-score |
| MLDATA-13 | Face crops, embeddings, or liveness frames sit in the same data class as account strings; the personal-data glossary has no body-derived subclass | **Body-derived identifiers are their own data class** — when a pipeline stores or compares descriptors of a person's body for identification or verification, give those fields a distinct class in the classification table carrying its own access, export, logging, and retention rules, because under the generic personal-data class they inherit account-string handling — soft delete, broad internal read, capture in logs and backups — for data the subject can never change after exposure (↔ sec [SECD-09] the identifier cannot be reissued once it leaks) | Which class do face templates sit in, and how do its access, logging, and retention rules differ from the class covering email addresses? | S·p | face-recognition-compulsory-visibility |
| MLDATA-14 | A template or enrollment table has no column for how the record was obtained, or the match path serves records where that column is null | **Capture basis is a column, not a policy page** — when a capture path can persist a matchable template, record on the record itself how it was obtained (affirmative enrollment, contractual, statutory, harvested) and make the match path refuse any record whose basis is null or unrecognised, because a basis held only in prose cannot be queried, so no one can enumerate which templates must be dropped when a subject withdraws or when a harvested source is later found unusable, and the withdrawal is executed by hand or not at all (↔ [MLDATA-04] the same per-record origin discipline, authorization rather than integrity) | Can the match path return a hit for a template whose capture-basis field is null? | S·w | face-recognition-compulsory-visibility |
| MLDATA-15 | A template store, gallery, or embedding index has no maximum-retention or expiry field and no disposal job; identity images sit in object storage or training staging with no lifecycle rule | **Every identity store carries a retention maximum** — when body-derived identifiers are persisted, set a maximum retention per declared purpose on the store, run an automated disposal job against it, and inventory every copy the identifiers reach (backup, staging, export), because without a bound the default lifetime is forever and the copies accumulate faster than any later deletion duty can be discharged by hand (deletion *propagation* is [RAG-10]; this rule supplies the expiry event that propagation runs against) | What is the maximum retention on this store, which job enforces it, and which copies are in the inventory? | S·w | face-recognition-compulsory-visibility |
| MLDATA-16 | A gallery or watchlist has no written membership rule — nothing states who is added, by what event, or what removes them | **A gallery needs an enrollment predicate** — when a store of identity templates can be searched, state the membership rule and enforce its bound in configuration (capture-source allowlist, enrollment-event source, membership expiry), because otherwise the gallery's population is whoever the cameras happened to see, no reviewer can enumerate who is searchable, and no subject ever leaves | Who is in this gallery, by what event did they enter, and what removes them? | S·p | face-recognition-compulsory-visibility |
| MLDATA-17 | A dataset card, release, or down-classification calls face images anonymous or non-personal because names, identifiers, or metadata were stripped | **Removing names does not de-identify a face** — when face-bearing data is released or moved to a lower data class on a de-identification claim, back the claim with a measured re-identification attempt (match the transformed data against a gallery of the same subjects and report the rate) or apply a transform that fails that match, because the identifier is carried by the images themselves and survives every field stripped from around them, so a name-stripped release is a searchable biometric set shipped under a claim its own contents falsify (↔ biz [CLM-04] the adjective is not the evidence) | What match rate did the transformed set score against a gallery of the same people? | S·r | face-recognition-compulsory-visibility |

## 2. EMB — Embedding spaces, normalization, versioning

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| EMB-01 | Vectors from different models, versions, or preprocessing enter one query or index | **One embedding space per comparison** — vectors are comparable only under the same model, preprocessing, and normalization contract; mixing spaces makes every similarity meaningless | Can every vector in this query name the same model contract? | B·w | designing-ml-systems |
| EMB-02 | A cluster or identity is represented only by an arithmetic mean of its observations | **Robust template, not a raw mean** — reject weak observations and compare a medoid / quality-weighted aggregate; one blurred or wrong frame otherwise moves the identity (↔ [EMB-04] identity-void samples move the representation) | Can one blurred or incorrect observation move the identity? | S·w | adaface |
| EMB-03 | A quality-aware embedding pipeline needs a quality signal at train or serve time | **Use a validated quality proxy** — take the signal from a proxy shown to track image quality; for a margin-trained face embedding use the **pre-normalization feature norm** ‖z‖ (the L2 norm before the vector is unit-normalized onto the hypersphere — after normalization it is identically 1), not class confidence alone | Is the quality signal validated, and taken before L2-normalization rather than from confidence? | S·w | adaface |
| EMB-04 | Identity loss trains on data with blur / crop / surveillance augmentation | **Quality-gate hard-example mining** — upweight hard samples only where the quality proxy is high and de-emphasize the extreme-hard low-quality ones; naïve hard mining fits unidentifiable images | Is hard-mining gated by quality, or applied globally? | S·w | adaface |
| EMB-05 | A feature input is a live TF-IDF, topic map, embedding table, or another model's output with no version pin | **Pin embedding and mapping versions** — consume a frozen, versioned copy and dual-eval before cutover; a consumer fitted to the old geometry breaks on a silent hot-swap (↔ [DRIFT-02], grph [GRPH-14] a derived map with no producer pointer is an unwalkable claim) | Which embedding/map version is this request on? | B·w | hidden-technical-debt-ml |
| EMB-06 | An identity decision is taken from a single-image classifier head or an unpaired embedding | **Recognition is a differential comparison** — require a feature-vector pair (probe–gallery) similarity against a threshold; classifier accuracy is not recognition fairness (↔ grph [GRPH-18] a non-pairwise name is the chaining failure one hop earlier) | Does identity acceptance use a pairwise similarity score against a threshold? | B·w | nist-frvt-demographics |
| EMB-07 | A multi-observation subject (track, burst, session) is matched by selecting one "best" observation — sharpest, most frontal, highest detector score — and discarding the rest | **Fuse the retained observations, do not match on one** — when a subject is observed repeatedly under one continuous association, gate the weak observations then compare an aggregate of what remains, because a single-exemplar selection stakes the decision on one observation's failure and throws away the pose and illumination variety that is the only thing compensating for weak individual evidence (↔ [EMB-02]: the robust aggregate is computed over the retained set — picking the sharpest single observation by an external quality criterion is not a robust aggregate) | Does the compare use an aggregate of the retained observations, or one selected observation? | S·w | video-to-video-face-surveillance |
| EMB-08 | The stored per-subject template is a variable-length list of per-observation vectors; index bytes and query cost grow with how long the subject was observed | **Template size independent of evidence count** — when observations per subject are unbounded, emit a fixed-size (or fixed-k) template as the indexed and first-stage-compared representation, because a representation that grows with evidence count makes storage and comparison scale with capture duration instead of gallery size, and interactive search dies as observations accumulate; put full pairwise comparison in a budgeted re-rank over a short candidate list (↔ eng [MODEL-02] which collections may grow without bound; [COST-07] cheap tier first, expensive path on the survivors) | Do template bytes and compare cost change if a subject is observed for twice as long? | S·w | video-to-video-face-surveillance |
| EMB-09 | Inputs of varying quality are compared by a raw distance or cosine against one threshold; how ambiguous each input is never enters the score | **Uncertainty-penalized similarity** — when compared inputs vary in quality, use a similarity that penalizes the ambiguity of *either* side so two ambiguous observations cannot produce a high score, because under degradation the two error directions move at once: different subjects observed badly converge toward each other while one subject observed well and badly diverges, so a deterministic distance reports geometry rather than evidence and the false accepts land exactly at the low-false-match end of the curve where the operating point lives (↔ [CAL-01] the per-stratum threshold is the other road to the same failure; [EMB-03] supplies the signal when no uncertainty output exists) | Can two low-quality observations of two different subjects produce a high similarity score? | S·w | handbook-face-recognition |
| EMB-10 | A subject template pools media whose per-source sample counts differ by orders of magnitude — one continuous recording's frames alongside individual captures — with no declared weighting | **Weight by independent capture, not by sample count** — when a template pools observations from sources with different capture rates, weight so one continuous capture contributes what one independent capture contributes, because equal per-sample weighting lets acquisition rate stand in for evidence: the longest recording decides the identity and the template describes that session's conditions rather than the subject (↔ [MLDATA-07] count unique subjects, not rows — this is the same count-is-not-coverage failure inside one template; [EMB-02] quality weighting does not fix it, since many sharp frames of one session still outvote every other source) | If one source contributed ten times more samples, would it decide the template? | S·w | janus-benchmark-c |

## 3. EVAL — Offline, pipeline, and user-level evaluation

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| EVAL-01 | A model metric is reported with no random / heuristic / human / production baseline | **Require offline baselines** — compare to random, zero-rule, a simple heuristic, a human, and the existing solution on the same split; without them the number is uninterpretable | What is Δ versus zero-rule and versus current production? | S·r | designing-ml-systems |
| EVAL-02 | Classes are imbalanced and only overall accuracy / F1 is reported | **Imbalance-aware metrics** — declare the positive/critical class and report per-class PR/ROC; a majority-constant model otherwise wins the gate | Does the champion beat a zero-rule baseline on the critical class? | B·r | designing-ml-systems |
| EVAL-03 | The training set is resampled (SMOTE / over- / under-) and metrics are read off the rebalanced set | **Never evaluate on the resampled distribution** — compute the ship-gate metric on natural / production prevalence; rebalanced metrics overfit a world that never ships | Is the ship-gate metric computed on an unrebalanced holdout? | B·r | designing-ml-systems |
| EVAL-04 | Only an overall metric is reported while mobile / geo / paid / minority cohorts exist | **Slice-based gate** — report and floor metrics per critical slice, and watch **Simpson's paradox** where the aggregate flips the within-slice ranking (↔ grph [GRPH-22] both refuse one global threshold where a verified partition is needed) | What is performance on each critical slice at n ≥ threshold? | B·r | designing-ml-systems |
| EVAL-05 | Output must not depend on a protected attribute or a strong proxy for one | **Invariance tests** — counterfactually swap (or exclude) the attribute and measure the output change; accuracy alone hides a discriminatory decision | Do race / name / gender swaps leave the decision unchanged? | B·r | designing-ml-systems |
| EVAL-06 | Training capture is controlled but production inputs are noisy | **Perturbation eval gate** — score candidates on a perturbed test suite matching production noise and pick the model that wins there, not on clean data | Did we choose the model that wins on perturbed, not only clean, test? | S·r | designing-ml-systems |
| EVAL-07 | Hyperparameter search or early-stopping reads the test-set metric | **Validation-only selection** — select on validation and report the test set exactly once; a tuning loop that can see test yields optimistic scores | Is the test loader unreachable from the tuning loop? | B·w | designing-ml-systems |
| EVAL-08 | A production multi-signal model changes a feature, hyper-parameter, sampling policy, or ensemble component, checked only on an aggregate metric | **CACE — measure slices before accepting a change** — in a multi-signal model, changing anything changes everything; require slice (and system-level) prediction deltas or a local win becomes a global loss | Which slices moved when we changed this one thing? | B·r | hidden-technical-debt-ml |
| EVAL-09 | The feature set grew with no leave-one-feature-out / ablation in the recent training report | **Ablate underutilized features** — run regular LOFO / grouped ablation and drop near-zero-lift features; a retained weak dependency is catastrophic brittleness when its source dies | Which features can we drop with no metric loss? | S·r | hidden-technical-debt-ml |
| EVAL-10 | The test set is drawn from the pool left over after active-learning selection | **Freeze the test set before active learning** — seal a random (plus out-of-distribution) test set before iteration 0 and never let active-learning selection train on it, or uncertainty sampling silently contaminates evaluation | Are test IDs sealed before iteration 0 and absent from all AL training? | B·p | human-in-the-loop-ml |
| EVAL-11 | Open-ended generation scored by exact match or a single gold string | **Open-ended tasks need the right scorer** — when many outputs are acceptable, drop sole exact match: prefer functional/oracle checks when the output is executable (code, SQL, tool calls), else a rubric, multiple references, or a validated preference/judge, because string equality measures wording, not task success (↔ biz [AIPX-02] the proxy moved, not the outcome) | Is the output executable (score functionally) or open prose (rubric/judge)? | S·w | ai-engineering |
| EVAL-12 | An LLM-as-judge gates releases without ever being checked against human labels | **Validate the judge against humans** — before an LLM judge gates decisions, measure its agreement with blinded human judgments on this task and re-validate on judge change, because an unvalidated judge steers selection and release with systematically wrong scores; anchor on a human gold set ([HITL-03]) | What is the judge's measured agreement with human gold on this task? | S·r | ai-engineering |
| EVAL-13 | A judge score is trended without a pinned judge model, prompt, rubric, decoding, seed, or order-handling protocol | **Pin the judging protocol** — version the judge model, prompt, rubric, decoding settings, random seed, and the order-randomization/counterbalancing protocol so scores are comparable across runs; pin the *protocol*, not a fixed presentation position for either candidate (which only stabilizes a systematic bias) (see [PROV-09]) | Are the judge's model, prompt, rubric, seed, and order-handling protocol pinned and versioned? | S·r | ai-engineering |
| EVAL-14 | A pairwise or scoring judge is used without counterbalancing candidate order or measuring its biases | **Counterbalance and measure judge bias** — run both A/B and B/A (or a balanced randomized protocol) and report disagreement and position-effect size, and likewise control for verbosity and self-preference (length-matched, no sole same-family self-scoring), because a single fixed order lets the first candidate win independently of quality and longer or same-family answers win regardless of merit (↔ ux [HAI-10] presentation order predicts the label unless shuffled and audited) | Were candidate positions counterbalanced, and the position, verbosity, and self-preference effects reported? | S·w | ai-engineering |
| EVAL-15 | A model is chosen mainly on public-benchmark scores | **Treat public benchmarks as contaminated** — treat public-benchmark performance as contaminated until proven otherwise and keep a private or time-separated eval, because training-set leakage makes memorization look like capability; keep a private or time-separated eval — the same contamination discipline validation-only selection uses against leakage ([EVAL-07]) | Is there a private or time-separated eval, or only public leaderboard scores? | S·p | ai-engineering |
| EVAL-16 | The shipped path is detect → associate → compare, but the reported identification metric is computed on ground-truth or hand-cropped inputs | **Upstream failure counts against the end-to-end metric** — when an earlier stage supplies the units the decider consumes, gate on at least one metric in which a subject the pipeline failed to detect or associate counts as a decision miss, because oracle inputs measure a system nobody deploys: upstream misses never reach the downstream service-level objective, and a detector regression ships invisibly behind an unmoved recognition number (↔ eng [TEST-10], [RAG-09], ux [UXR-01] keep stage-isolated numbers for *attribution* — this row forbids substituting them for the operational claim) | Does a subject the detector missed count as an identification error in the ship gate? | S·r | janus-benchmark-c |
| EVAL-17 | Production pools many observations per subject into one comparison unit, but the evaluation compares individual observations | **Evaluate at the unit operations aggregates** — when the deployed comparison unit is a pooled multi-observation template, build the evaluation's probe and gallery units the same way and run the deployed pooling path, keeping a single-observation protocol only for claims about individual capture conditions, because a per-observation protocol measures a matcher the product never calls: pooling both hides single-observation difficulty and supplies evidence the tested configuration never had, so the number moves in an unknown direction | Are the evaluation's probe and gallery units built by the production pooling path? | S·p | janus-benchmark-c |
| EVAL-18 | An identification system that must reject non-enrolled subjects is gated on rank-based metrics or training-set classification accuracy, and every evaluation probe has a mate in the gallery | **Non-mated probes and a score threshold, not rank** — when the query population contains subjects absent from the gallery, include non-mated probes and report the false-negative identification rate (mated searches returning no mate above the threshold) against false-positive identifications at a fixed score threshold, because a rank metric is computed over a list that always contains an answer: it cannot express "no one here", so a system that names every stranger scores well and nothing in the gate has met a stranger (↔ [CAL-02] designs the reject path; this row measures it; [EVAL-03] the evaluated prevalence of non-mates must be the production one) | How many evaluation probes have no mate in the gallery, and at what score threshold are their outcomes reported? | S·r | janus-benchmark-c |
| EVAL-19 | A false-positive rate is normalized by a count the system under test produces — its own searches, detections, or candidate rows | **Do not divide an error by a volume the system controls** — when the measured system can change the denominator of its own error rate, report the error as an absolute count against a fixed external exposure (per hour, per enrolled subject, per real query) or fix the denominator outside the system, because an endogenous denominator pays for producing more of the divisor: loosening the upstream stage mints extra non-mated searches, the reported rate falls, and every operator's workload rises (↔ ux [UXR-07] the denominator must be predefined — this row adds *by whom*; eng [TEST-11], biz [OPS-01] the metric pays for the behaviour it rewards) | Does this rate improve if the system emits more low-quality candidates? | S·r | janus-benchmark-c |
| EVAL-20 | A comparison system reports one accuracy number while the two compared sides come from different capture regimes (enrolled portrait vs field capture, clean vs degraded) | **The slice is a cell of the pair, not a stratum of one side** — when a decision compares two inputs, stratify by the cell (query regime × enrolled regime) and measure every cell the product will run, because the two sides are not interchangeable: degrading the enrolled side is a different failure from degrading the query side, and a system measured against clean enrolments carries no evidence at all for the cell where both sides are degraded — which is usually the deployment (↔ [EMB-06] a comparison has two sides, so its slices do too; [EVAL-04] and [FAIR-05] stratify one side each) | Which query × enrolled regime cell does the ship-gate number measure, and which production cells are unmeasured? | S·r | handbook-face-recognition |
| EVAL-21 | The product returns a ranked candidate list for a person to scan, but the ship gate is pair-verification accuracy, equal-error rate, or top-1 correctness | **Gate the depth the human actually reads** — when the output contract is a ranked list an operator inspects, gate on rank-sensitive recall and precision over the inspected depth, because pair accuracy does not predict ranking order and top-1 optimizes the wrong loss for this consumer: interleaved false candidates are cheap (the human discards them) while a correct candidate buried below the read depth is a total miss (↔ sec [SEC-05], ux [HAI-12] own the *gate* — a human decides before the act; this row owns the *metric* that gate implies; [EVAL-11] choose the scorer the task actually needs) | How deep does the operator read, and is the metric computed over that depth? | S·p | video-to-video-face-surveillance |
| EVAL-22 | A release gate is decided on an offline metric, and no experiment has ever measured what a movement in that metric does to the product metric | **Validate the proxy by degrading it** — before an offline metric may gate a release, ship a deliberately degraded model to a small share of live traffic and measure whether the product metric moves in the predicted direction and magnitude, and re-run that measurement when the objective or the traffic mix changes, because an offline score never observed moving a product metric is an unvalidated proxy and optimizing it can buy live quality loss (the product metric's sovereignty is biz [AIPX-02]; mutating the path to prove the detector is live is eng [TEST-15]) | Which experiment showed a known-worse model moving the product metric? | J·p | ml-test-score |

## 11. VSEG — Temporal structure of a media timeline

Where `TRACK-*` follows one entity *across* frames, `VSEG-*` cuts the timeline *into* units — shot, scene, keyframe. Different problem, different failure modes: a tracker loses an identity, a segmenter puts a boundary in the wrong place or invents one. Rules here fire on a shot-boundary detector, a scene-grouping step, a keyframe/thumbnail selector, or a schema that names a `shot`, `scene`, or `chapter`. Boundary quality is measured with imbalance-aware per-class precision/recall `[EVAL-02]` sliced by transition class `[EVAL-04]`, counting a **false split** (one shot reported as two) separately from a **false merge** (two shots reported as one).

| ID | Trigger | Rule | Answers | T·P | Src |
| --- | --- | --- | --- | --- | --- |
| VSEG-01 | Schema exposes one flat `clip`/`segment` unit, or a `scene` table whose rows are 1:1 with detected cuts | **Shot and scene are different units** — when cataloguing edited video, type the timeline as program ⊃ scene ⊃ shot ⊃ frame with separate identifiers, and derive scenes by *grouping* shots rather than by detecting more cuts, because a shot is a physical capture boundary and a scene a semantic one; collapsing them leaves the schema unable to express a dialogue exchange that spans a dozen shots (Gao Preface, §1.1, §5.1 — Table 5.1 counts ~1000 shots against ~30 scenes per film) | Are shot and scene distinct typed units, with scenes derived by grouping? | S·p | video-cataloguing-gao |

| VSEG-03 | Candidate boundaries come straight from a frame-difference threshold, or frames are subsampled for speed with no refinement pass | **Filter motion-caused false boundaries** — when a pan, zoom, or large object motion spans far longer than a gradual transition, classify the resulting spike as a motion-caused false boundary and suppress it; when subsampling for speed, refine each candidate interval at full frame rate before accepting it, because sustained motion is not an edit and bare subsampling both drops real boundaries and invents motion artefacts (Gao §3.4–3.6, Table 3.3) | Are motion false positives measured and filtered, and is every subsampled candidate refined? | S·r | video-cataloguing-gao |

| VSEG-06 | Keyframes produced by clustering are stored or displayed in cluster order | **Restore chronological order after clustering** — when keyframes come from a clustering step, re-sort by timestamp before storage or display, because cluster order drops the chronology a storyboard needs to be readable (Gao §4.3) | Is the keyframe list monotone in timestamp? | J·r | video-cataloguing-gao |

## 15. Cross-lexicon links

Machine-learning systems sit downstream of the graph lexicon and upstream of the human review surface, so its rules border several others:

- **Graph substrate.** This lexicon feeds and gates `[GRPH-17..23]`. `CAL-01` (calibrate per quality stratum) is the embedding-and-threshold form of `[GRPH-23]` (per-quality-stratum edge policy); `EVAL-04` (slice gate, Simpson's paradox) is `[GRPH-22]` (partition-plus-verification) in metric form; and a mis-calibrated `FAIR-03`/`CAL-04` accept is exactly the wrongly-merged identity `[GRPH-18]` warns about — the chaining failure, one embedding threshold upstream.
- **Provenance & authority.** `PROV-01/05/07` meet `[GRPH-14]` (provenance as a directed citation graph) and engineering `[DATA-14]` (one authority owns the lineage). All four run the same audit: walk backward from the claim; if the walk breaks, the claim breaks. This is [Principle 9] wearing an ML uniform.
- **Security.** Biometric `FAIR`/`CAL` rules border `[SEC-05]` (a human gate on irreversible automated action) and `[SEC-10]` (pin the exact model/dataset revision). `CAL-06` (no demographic-conditioned threshold) is a security rule stated as fairness; `PROV-05` (declare prediction consumers) is `[SEC-04]` (least privilege) on the prediction surface.
- **Release & claims.** `EVAL`/`FAIR` ship-gates are `[RLSE-04]` (no undesigned state ships) and `[PROD-09]` (author the contract the gatekeeper reads) for models; `PROV-02` and every disaggregated-eval rule are `[CLM-05]` (a claim needs inspectable evidence) — "accurate," "fair," "safe for X" require the card's quantitative analyses.
- **Debugging.** `DRIFT-04` (live prediction-bias monitor) is the first diagnostic layer `[DBG-*]` reaches for before deep model debugging; `EVAL-08` (CACE slices) is where a "local" fix is caught changing something global.
- **Tracking builds on embedding and calibration.** `TRACK-*` is the temporal layer above `EMB` and `CAL`: `TRACK-07` (predict then associate) and `TRACK-05` (gated template update) are the video form of `EMB-06` (recognition is a differential comparison) and `[GRPH-18]` (the chaining failure) run across frames — a bad frame-to-frame link is a wrongly-merged identity in time. `TRACK-06`/`TRACK-07` leave an ambiguous association `unknown` exactly as `CAL-02` prescribes, and `TRACK-05` (gated template update) is `EMB-02` (robust template) plus `DRIFT-01` under drift and occlusion. Drawing the tracks is interaction `[VIZ-13]`.
- **Cost is the budget beside serving and evaluation.** `COST-*` cross-links but does not restate engineering `[PERF-01]`/`[PERF-02]` (percentiles, tail-latency), `[PERF-06]` (measure, don't guess), or `[OBS-*]` (SLO/alert machinery); it owns the ML framing — `COST-04` (cost per accepted output) and `COST-01` (ML-layered SLOs) are `[Principle 4]` (you get the number you pay for) on the serving budget, `COST-12` (staleness is a budget item) prices the retrain that `DRIFT-01` forces, and `COST-10` (graceful degradation) is `SERVE-02` + `CAL-02` as an availability control.

## 16. Consumption

Referenced by ID at the decision point, never read front-to-back. `MLDATA-*` fire at dataset construction; `EMB`/`CAL`/`FAIR` fire in embedding, recognition, and evaluation code; `EVAL-*` at the offline ship gate; `PROV-*` at model release and at any store that persists a derived output; `DRIFT`/`SERVE` in production monitoring and the serving path; `HITL-*` wherever humans label, correct, or adjudicate model output; `TRACK-*` on a video tracking config, a track-log, or an ID-switch/fragmentation metric; `COST-*` on an SLO doc, a latency/cost dashboard, a capacity plan, or a retraining config, at plan and review time; `FM-*` on a prompt/template, a system-instruction or few-shot change, an adaptation/fine-tuning plan, or an agent-loop definition; `RAG-*` on a corpus/chunker/index, a retriever/reranker, or a context-construction change. The single rule to cite on any pipeline that accepts an identity or **names an entity** from a similarity score is `FAIR-03` + `CAL-02` + `[GRPH-18]` — the three standing between a similarity score and a wrongly-named person. Retrieval routing (per the expansion brief): embedding or recognition code → `EMB + CAL + FAIR + PROV + GRPH`; a model replacement or weight change → `EVAL + CAL + PROV + SEC`; a video tracker → `TRACK + EMB + CAL`; a serving-cost or SLO review → `COST + SERVE + eng [PERF/OBS]`; a prompt or generation-contract change → `FM + EVAL + PROV + SEC`; a RAG corpus/index change → `RAG + EMB + DRIFT + PROV + SEC + COST + OBS`; an agent loop → `FM + RES + COST + OBS + PROV + SEC`. A consuming harness inlines the trigger plus the question by phase tag and cites the ID; the argument stays in `distilled/ml-systems/`.
